# 🏢 HR Analytics Dashboard — Compensation & Benefits

Веб-застосунок на Streamlit для аналізу зарплат, бонусів, пільг і повного компенсаційного пакета співробітників. Дозволяє швидко побачити структуру компанії, гендерний розрив в оплаті праці та порівняти внутрішні зарплати з ринковими.

## Ідея проєкту

HR-команді часто потрібно відповісти на кілька простих, але важливих запитань: скільки компанія платить людям насправді (з урахуванням пільг, а не лише окладу), чи є розрив в оплаті між чоловіками та жінками, та чи відповідають зарплати ринку. Зазвичай це робиться вручну в Excel. Цей застосунок автоматизує такий аналіз і показує його у вигляді інтерактивного дашборду з фільтрами.

## Результат

Інтерактивний веб-дашборд із 6 сторінками та живими фільтрами (відділ, посада, діапазон зарплати, тип пільги, наявність бонуса). Дані оновлюються миттєво при зміні фільтрів, без перезапуску застосунку.

## Можливості (сторінки)

| Сторінка | Що показує |
|---|---|
| 🏠 Головна | Загальний огляд: кількість співробітників, середні показники, структура по відділах |
| 📊 Ключові показники | KPI-картки, загальний бюджет компенсацій, структура пакета (база / бонус / пільги), охоплення пільгами, експорт у CSV/Excel |
| 🎁 Аналіз бонусів та компенсацій | Статистика бонусів, охоплення пільгами по відділах, 5 типів графіків (стовпчиковий, лінійний, кругова діаграма, теплова карта, бульбашковий) |
| 👩‍🦰👨 Гендерна аналітика | Порівняння середньої/медіанної зарплати за статтю, розподіли, гендерний розрив |
| 🏆 Рейтинг співробітників | TOP-N співробітників за повним компенсаційним пакетом |


## Формат файлу compensation.csv

| Колонка | Тип | Опис |
|---|---|---|
| `id`, `employee_id` | int | ідентифікатор співробітника |
| `name` | текст | ім'я |
| `department` | текст | відділ |
| `Role`, `Role_ua` | текст | посада (англ. / укр.) |
| `base_salary` | число | базова зарплата, грн |
| `bonus` | число | бонус, грн |
| `health_insurance` | 0/1 | чи є медстрахування |
| `sport` | 0/1 | чи є спортивна пільга |
| `remote_allowance` | 0/1 | чи є виплата на remote |
| `gender` | текст | стать: `Жінка` / `Чоловік` |
| `performance_score` | число | оцінка ефективності (опційно) |

`total_compensation` у файлі **немає** — він рахується автоматично:

```
total_compensation = base_salary + bonus + (сума вартості активних пільг)
```

Вартість кожної пільги — фіксована константа в коді (`BENEFIT_COSTS`), а не значення з файлу.

## Формат файлу market_salaries.json

Це підготовлений вручну (або одного разу вивантажений з Work.ua/DOU) набір ринкових зарплат — `parser.py` лише читає його, без live-запитів до сайту:

```json
{
  "updated": "2026-01-15",
  "data": {
    "Програміст": 45000,
    "HR-менеджер": 32000
  }
}
```

Ключі в `data` мають **точно** збігатися з рядками з колонки `Role_ua` у `compensation.csv` (включно з регістром) — інакше для цієї посади буде показано «Не вказано». Якщо файл відсутній або пошкоджений, `get_market_data` не падає, а повертає `salary: "Не вказано"` із текстом помилки.

## Технічні рішення, варті уваги

Дві речі, які не очевидні з першого погляду, але показують усвідомлений підхід до архітектури:

1. **Перемикач джерела ринкових даних з однаковою структурою відповіді.** Сторінка "Аналіз ринку" однаково працює і з підготовленим набором даних (`market_salaries.json`, підписаний як "Work.ua/DOU"), і з синтетичним "власним дослідженням" (`np.random`) — обидва джерела повертають дані в одному форматі, тож UI не залежить від того, звідки прийшли цифри. Завдяки цьому демо не залежить від інтернету чи доступності зовнішнього сайту.


2. **Стійкий до помилок парсер.** `get_market_data` обгортає читання файлу в `try/except`: якщо `market_salaries.json` відсутній, пошкоджений або не містить потрібної посади — функція не кидає виняток, а повертає той самий за структурою словник із `"salary": "Не вказано"`. Завдяки цьому `page_market` ніколи не падає, а просто показує повідомлення "Недостатньо даних".

3. **Кастомний UI без виходу за межі Streamlit.** Сторінка "Ключові показники" не використовує стандартний `st.metric` (обмежений у стилізації), а власні HTML/CSS-картки: hero-блок з градієнтом і смужкою "структура компенсації" (база/бонус/пільги у %), прогрес-бари охоплення пільгами. Усе це — звичайний `st.markdown(..., unsafe_allow_html=True)`, без додаткового фронтенд-фреймворку.



## Нюанси та обмеження

- CSV читається при **кожній** взаємодії з фільтрами (без `st.cache_data`) — для дуже великих файлів варто додати кешування.
- Сторінка "Гендерна аналітика" розрахована на рівно два значення: `Жінка` і `Чоловік`.
- `market_salaries.json` — це статичний файл, не live-запит до сайту, тож інтернет для роботи застосунку не потрібен. Але дані в ньому потрібно оновлювати вручну, і ключі (`Role_ua`) мають точно збігатися з посадами у `compensation.csv`.
- "Власне дослідження" генерує випадкові зарплати лише для фіксованого списку з 9 посад (`Програміст`, `HR-менеджер`, `Маркетолог` тощо). Якщо обрана посада не входить у цей список — порівняння буде недоступне («Недостатньо даних»), це очікувана поведінка, а не помилка.
- Верхній автоматично згенерований список сторінок Streamlit прихований через CSS — вся навігація реалізована вручну в блоці "Меню" (`st.sidebar.radio`), не плутати одне з іншим.
